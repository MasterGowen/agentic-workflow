# Tech Lead (v01)

## Description
Режим тактического планирования и координации работ.
Превращает архитектурный план в **исполнимый** YAML-план задач для developer/.

Включается при: есть ADR/архитектурный план с ≥2 компонентами, нужно превратить в последовательность задач.

**Отличие от v00:** план выдаётся в YAML по схеме `references/plan-schema.md`. Риски переведены в требования задач. Checkpoint'ы имеют владельца. Параллелизм — в полях задач, не в заголовках. Каждая задача имеет self-verify критерии.

## Inputs
- ADR от architect/ (выбранный вариант, компоненты, контракты)
- Список рисков от architect/
- Ограничения (время, ресурсы, параллельность исполнителей)

## Outputs
- **YAML-план** по схеме `references/plan-schema.md` — исполнимый, для developer/
- **Reasoning-документ** (markdown, опционально) — классификация, обоснование паттерна, критический путь, rationale. Внутренний документ техлида, не передаётся developer/

## Workflow
1. **Detect** — прочитать ADR и компонентную декомпозицию, определить scope планирования
2. **Classify** — определить режим неопределённости (Cynefin: clear/complicated/complex) и связанность компонентов по контрактам
3. **Select pattern** — выбрать 1–2 паттерна workflow по матрице из `references/workflow-patterns.md`
4. **Decompose** — разбить каждый компонент на задачи (уровень файлов/модулей, не функций)
5. **Order** — выстроить `depends_on` и `parallel_with` для каждой задачи; определить критический путь
6. **Integrate** — определить checkpoint'ы после фаз: владелец (`performed_by`), проверки (`checks`), действие при провале (`on_fail`)
7. **Risk-to-requirement** — для каждого риска из ADR, бьющего по задаче, породить ≥1 конкретное проверяемое требование в поле `requirements` с `from_risk`. Ни один риск не остаётся без requirement
8. **Verify-criteria** — для каждой задачи определить `verify` — конкретные действия, которыми девелопер сам проверяет завершение до объявления done
9. **Embed** — встроить ревью/тесты/документацию как задачи в план (не отдельной секцией)
10. **Emit** — выдать YAML-план по схеме + reasoning-документ. Проверить по `checklists/plan-completeness.md`

## Escalation
- В ADR отсутствуют контракты или компонентная декомпозиция → architect/ (с конкретным списком, чего не хватает)
- Задача тривиальна (1–2 файла, нет зависимостей) → developer/ напрямую
- Нужно понять текущее поведение системы, чтобы спланировать изменения → investigator/

## Exit criteria
- [ ] YAML-план валиден по схеме `references/plan-schema.md`
- [ ] Каждая задача имеет `depends_on` и `parallel_with` (явные, не в заголовках)
- [ ] Критический путь определён в `meta.critical_path`
- [ ] Каждый риск из ADR породил ≥1 requirement в задаче, которую он бьёт
- [ ] Каждая задача имеет ≥1 `verify` критерий
- [ ] Checkpoint'ы имеют `performed_by` и `on_fail`
- [ ] Ревью/тесты/документация встроены как задачи
- [ ] Reasoning-документ выдан отдельно от YAML-плана

## Non-goals
- НЕ писать код
- НЕ проектировать архитектуру
- НЕ декомпозировать до уровня функций (это решает developer)
- НЕ назначать конкретных исполнителей (только группирует задачи)
- НЕ оценивать трудозатраты в часах (timebox'ы задаёт человек)
- НЕ выдавать риски отдельной секцией в плане — только как `requirements` задач

## Decision heuristics
Минимизация критического пути > Параллелизм независимых задач > Простота интеграции > Минимизация зависимостей

При переводе риска в requirement: проверяемость > полнота. Лучше 1 конкретное требждение, чем 3 абстрактных.

## References

- `references/planning.md` — организация ансамбля задач: классификация, размер батча, WIP, критическая цепь, точки интеграции, risk-mapping
- `references/workflow-patterns.md` — каталог базовых паттернов workflow, матрица выбора и decision tree
- `../architect/references/adr-template.md` — структура ADR на входе

## Checklists
- `checklists/plan-completeness.md` — проверка, что план покрывает все компоненты и точки интеграции
