# Workflow Patterns — Базовые паттерны организации работ

Каталог паттернов для шага `Select pattern` в workflow tech-lead. Паттерны можно комбинировать (не более 2 одновременно — иначе теряешь прозрачность).

См. также `planning.md` для организации ансамбля задач (размер батча, WIP, критическая цепь, risk-mapping).

## 1. Linear / Sequential
- **Структура**: A→B→C, каждая задача зависит от предыдущей
- **Когда применять**: сильная связанность, контракты нестабильны, нельзя распараллелить
- **Антипаттерн**: применять при наличии независимых задач — теряешь время
- **Пример**: миграция схемы БД → код → тесты — каждый шаг требует предыдущего

## 2. Fan-out / Fan-in
- **Структура**: 1 декомпозиция → N независимых задач → 1 интеграция
- **Когда применять**: контракты заранее зафиксированы, низкая связанность, задачи однотипны
- **Антипаттерн**: применять при нестабильных контрактах — интеграция превратится в переделку
- **Пример**: 5 эндпоинтов API по одному контракту — каждый агент берёт эндпоинт

## 3. Wave / Batched sync
- **Структура**: параллельные группы, синхронизация в «волнах» между группами
- **Когда применять**: частичные зависимости, нужна периодическая сборка
- **Антипаттерн**: делать волны слишком крупными — теряешь преимущество параллелизма
- **Пример**: бэкенд и фронтенд параллельно, синхронизация по контрактам каждые N задач

## 4. Pipeline
- **Структура**: стейджи с хендоффами, каждый стейдж берёт единицу работы
- **Когда применять**: повторяющаяся однотипная работа, разные скиллы на стейджах
- **Антипаттерн**: применять для уникальных задач — оверхед на стейджи не окупается
- **Пример**: код → ревью → тесты → документация — каждая задача проходит стейджи

## 5. Spike + Implement
- **Структура**: 1 исследовательская задача → реализация по её результатам
- **Когда применять**: высокая неопределённость, нужно узнать до планирования
- **Антипаттерн**: делать spike слишком длинным — превращается в бесконечное исследование
- **Пример**: неизвестно, какой библиотекой парсить X → spike на 1 задачу → реализация

## 6. Set-based concurrent
- **Структура**: N параллельных подходов, выбор поздно
- **Когда применять**: несколько жизнеспособных вариантов, ошибка выбора дорогая
- **Антипаттерн**: применять когда вариант один — трата ресурсов
- **Пример**: 2 прототипа хранилища параллельно → бенчмарк → выбор

## 7. Strangler Fig
- **Структура**: постепенная замена старой системы по границам
- **Когда применять**: миграция legacy, big-bang невозможен
- **Антипаттерн**: применять когда старую систему можно выкинуть целиком
- **Пример**: перенос эндпоинтов по одному на новый сервис, старый работает параллельно

## 8. Trunk + feature branches
- **Структура**: параллельные ветки от main, merge обратно
- **Когда применять**: независимые фичи, есть CI/CD
- **Антипаттерн**: длинные ветки — накопление конфликтов
- **Пример**: 3 независимые фичи в 3 ветках, merge по готовности

## Матрица выбора паттерна

| Неопределённость \ Связанность | Тонкие контракты | Толстые контракты |
| --- | --- | --- |
| Clear | fan-out | linear |
| Complicated | spike → fan-out | spike → linear |
| Complex | set-based | wave + buffer |

Матрица даёт базовый паттерн. Особые случаи (миграция, повторяющаяся работа, независимые фичи) — см. decision tree ниже.

## Decision tree — особые случаи
- Миграция legacy → **strangler fig**
- Несколько жизнеспособных вариантов, дорогая ошибка выбора → **set-based concurrent**
- Повторяющаяся однотипная работа, разные скиллы → **pipeline**
- Независимые фичи + CI/CD → **trunk + feature branches**
- Частичные зависимости между группами → **wave**

## Комбинирование
- Паттерны можно комбинировать: spike → fan-out → pipeline на стейджах ревью
- Не комбинировать больше 2 — теряешь прозрачность плана
- Если комбинация не описывается двумя паттернами — задача, скорее всего, требует декомпозиции на подпланы
