# Planning — Организация workflow задач

В отличие от `../architect/references/planning.md` (декомпозиция одной задачи), здесь — про ансамбль: как организовать несколько задач в workflow.

## Классификация задачи
- Определи режим неопределённости (Cynefin):
  - **Clear** — известны и проблема, и решение → linear
  - **Complicated** — проблема известна, решение требует анализа → spike + implement
  - **Complex** — неизвестны ни проблема, ни решение → probe-sense-respond, set-based
- Определи связанность компонентов по ADR:
  - **Тонкие контракты** (data-only, стабильные интерфейсы) → параллелизм возможен
  - **Толстые контракты** (поведенческие, меняющиеся) → sequential или wave
- Режим и связанность вместе определяют паттерн — см. матрицу в `workflow-patterns.md`

## Размер батча
- Маленький батч: короче петля обратной связи, меньше риск, но больше оверхед на координацию
- Большой батч: меньше интеграций, но дольше узнаёшь о проблеме
- U-кривая: оптимум там, где сумма cost-of-batch + cost-of-delay минимальна
- Эвристика: батч должен быть таким, чтобы интеграция проходила за один присест
- Не дробить до атомов — оверхед на координацию превысит выигрыш

## WIP-лимиты
- Little's Law: cycle time = WIP / throughput
- Больше параллельных незакрытых задач = дольше цикл, больше переключений
- Ограничивай параллелизм даже если теоретически можно больше
- Эвристика: не более 3–4 параллельных задач на одну точку интеграции

## Критический путь и критическая цепь
- **Критический путь** (PERT): самая длинная последовательность зависимостей задач
- **Критическая цепь** (Goldratt): критический путь + ресурсные ограничения
- Считай цепь, а не только путь — ресурсный конфликт может удлинить реальный цикл
- Буфер кладётся туда, где риск по цепи, а не равномерно
- Буфер на цепь в целом (project buffer) эффективнее, чем на каждую задачу (feeding buffers) — кроме случаев явного ресурсного конфликта

## Точки интеграции
- Используются для координации, НЕ включаются в бэклог для developer
- Частота интеграции = функция риска рассинхрона
- В каждой точке интеграции — контракт-проверка: реальные интерфейсы против ожидаемых из ADR

## Risk-mapping
- Используется для управления, НЕ включается в бэклог для developer
- Возьми риски из ADR
- Наложи на критическую цепь: какой риск бьёт по какой части
- Для каждого High-риска на цепи — буфер или fallback
- Риск вне цепи можно парировать без буфера

## Обратимость решений
- Оцени обратимость каждой точки интеграции
- Дорогой откат → откладывай коммит, держи варианты открытыми (set-based concurrent)
- Дешёвый откат → можно идти sequential без буферов
- Необратимые точки (миграция данных, изменение публичного API) — отдельная задача-проверка перед коммитом

## Формат выходных документов

Tech-lead создаёт **два документа**, а не один:

### 1. Техническая спецификация (`NN_tech_spec_<name>.md`)
- Модель данных: ERD, Qdrant payload, SQLite-схема, индексы
- Интерфейсы модулей: class-диаграммы с сигнатурами методов
- Sequence-диаграммы pipeline'ов
- MCP API: таблица всех инструментов
- План миграции
- Конфигурация
- Файловая структура
- Принципы, которые developer не должен нарушать

### 2. Бэклог задач (`NN_backlog_<name>.md`)
- Задачи сгруппированы по фазам
- Каждая задача: что сделать, детали, зависимости, файлы, критерии, границы, тесты, документация
- Граф зависимостей (ASCII)
- Критический путь

**В бэклоге НЕТ:** рисков, чек-листов, архитектурных рассуждений, точек интеграции, cognitive review.
