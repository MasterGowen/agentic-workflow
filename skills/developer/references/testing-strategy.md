# Testing Strategy — Методология тестирования

## Пирамида тестов
- **Unit-тесты** (основание) — изолированные, быстрые, покрывают бизнес-логику
- **Integration-тесты** (середина) — взаимодействие компонентов, БД, внешние сервисы
- **E2E-тесты** (вершина) — критические пользовательские сценарии, минимум

## Что покрывать
- Бизнес-логика: 100% unit-тестами
- API-эндпоинты: integration-тестами с тестовой БД
- Граничные случаи: null, пустые коллекции, крайние значения
- Обработка ошибок: каждый except-блок должен быть покрыт
- Регрессии: на каждый найденный баг — тест

## Fixtures
- Использовать factory-функции для тестовых данных
- Фикстуры должны быть минимальными и самодокументированными
- Избегать глобального состояния между тестами

## Mocking
- Мокать только внешние границы (HTTP, БД, файловая система)
- Не мокать внутренние сервисы — использовать реальные имплементации
- Предпочитать fakes вместо mocks, когда возможна простая in-memory реализация

## Запуск
- Тесты должны запускаться одной командой
- CI должен прогонять все тесты на каждый PR
- Медленные тесты маркировать и запускать отдельно
