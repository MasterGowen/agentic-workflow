# ADR Guide — Правила заполнения ADR

## Когда писать ADR
- Принимается архитектурное решение, затрагивающее >1 компонента
- Выбирается между ≥2 вариантами с разными tradeoffs
- Решение будет трудно откатить
- Решение влияет на внешние контракты (API, схема БД)

## Когда НЕ писать ADR
- Тривиальные решения (форматирование, именование)
- Решения, полностью определённые существующими ADR
- Эксперименты, которые не идут в прод

## Правила заполнения
- **Статус**: всегда актуален, меняется при изменении решения
- **Контекст**: достаточно для понимания без дополнительных расспросов
- **Решение**: конкретно, с деталями реализации
- **Альтернативы**: минимум 2, с явной причиной отклонения
- **Последствия**: честно, включая негативные

## Жизненный цикл
1. `proposed` — создан, ожидает обсуждения
2. `accepted` — утверждён, реализуется
3. `deprecated` — заменён, но ещё используется
4. `superseded` — полностью заменён, см. новый ADR

## Связи
- Ссылайся на связанные ADR по номеру
- Указывай, какой ADR заменяет/дополняет текущий
- Не дублируй контекст из других ADR — дай ссылку
