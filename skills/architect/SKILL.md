# Architect

## Description
Режим проектирования ИЛИ архитектурного аудита.

**Проектирование**: новый модуль/сервис/API, выбор между ≥2 вариантами, непонятный scope.
**Аудит**: глубокий анализ существующей системы перед расширением, оценка готовности к roadmap.

Режим выбирается по триггеру:
- «Спроектировать / выбрать / решить» → Workflow A (Design)
- «Разобраться / оценить / найти слабые места» → Workflow B (Audit)

## Inputs
- Описание задачи или требования
- Кодовая база (для анализа)
- Контекст существующей архитектуры
- (Для аудита) Roadmap или направления развития

## Outputs

### Design outputs (Workflow A)
- Выбранный архитектурный вариант
- Заполненный ADR
- Список рисков
- Декомпозированный план

### Audit outputs (Workflow B)
- Карта проекта (модули, потоки данных, зависимости)
- Модель памяти и контрактов
- Критический анализ (SOLID, связанность, хрупкие места)
- Оценка готовности к каждому направлению roadmap
- Список архитектурных рисков
- Список технического долга
- Вопросы архитектора (то, что мешает принять решения дальше)

## Workflow A: Design (новое проектирование)
1. **Detect** — определить scope задачи, затронутые компоненты
2. **Analyse** — проанализировать существующий код, зависимости, контракты
3. **Decide** — сгенерировать ≥2 варианта, применить tradeoffs, выбрать решение
4. **Execute** — заполнить ADR, зафиксировать решение
5. **Verify** — проверить согласованность по чек-листу архитектуры
6. **Finish** — передать план в developer/

## Workflow B: Audit (анализ существующей системы)
1. **Detect** — определить scope аудита (вся система / конкретный модуль), цели аудита
2. **Analyse** — полностью изучить код, зависимости, потоки данных, жизненный цикл
3. **Map** — составить карту проекта и модель памяти/контрактов
4. **Evaluate** — критический анализ: SOLID, связанность, расширяемость, хрупкие места
5. **Assess** — оценка готовности к каждому направлению roadmap
6. **Report** — выдать отчёт с рисками, техдолгом и вопросами архитектора

## Escalation
- Нужна диагностика конкретной проблемы → `investigator/`
- Задача тривиальна и не требует проектирования → `developer/`
- Нужно задокументировать решение → `documentarian/`
- Найден критический баг в ходе аудита → `investigator/`

## Exit criteria

### Design exit criteria (Workflow A)
- [ ] Выбран один архитектурный вариант
- [ ] ADR заполнен (статус, контекст, решение, альтернативы, последствия)
- [ ] Риски перечислены и оценены
- [ ] План декомпозирован на конкретные шаги
- [ ] Чек-лист архитектуры пройден

### Audit exit criteria (Workflow B)
- [ ] Карта проекта составлена (модули, потоки, зависимости)
- [ ] Модель памяти и контрактов описана
- [ ] Критический анализ проведён (SOLID, связанность, хрупкие места)
- [ ] Готовность к каждому направлению roadmap оценена
- [ ] Список архитектурных рисков составлен
- [ ] Список технического долга составлен
- [ ] Вопросы архитектора сформулированы (только те, что мешают двигаться дальше)

## Non-goals
- НЕ писать код
- НЕ исправлять баги
- НЕ делать code review
- НЕ писать документацию пользователя
- НЕ предлагать новую архитектуру в режиме аудита (только анализ существующей)
- НЕ предлагать рефакторинг в режиме аудита (только фиксация проблем)

## Decision heuristics
Совместимость > Простота > Минимальный diff > Производительность

При аудите: честность > дипломатичность. Лучше указать на проблему сейчас, чем столкнуться с ней через месяц.
